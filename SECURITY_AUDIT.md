# Аудит безопасности Vault PLM (2026-05-03)

Документ фиксирует выявленные риски и принятые меры. Проект — веб-клиент на Supabase (Auth + RLS + Edge Functions).

## Критические (исправлено в коде / миграциях)

### 1. Edge Function `vault`: неаутентифицированные чтения через service role

**Риск:** Маршруты `GET /versions`, `GET /nodes`, `GET /audit`, `GET /backups`, `GET /integrity-check` выполнялись с клиентом `service_role` без проверки JWT. Любой запрос без авторизации мог выгрузить метаданные всего хранилища.

**Меры:** Все маршруты требуют валидный access token; для чтений используется клиент с JWT пользователя (`createUserClient`), чтобы действовали политики RLS.

### 2. «Проверка» JWT без подписи

**Риск:** `extractUserId` декодировал payload без проверки подписи — возможен произвольный `sub` и обход авторизации на `POST`.

**Меры:** Используется `supabase.auth.getUser(jwt)` с anon-ключом (проверка подписи на стороне Supabase).

### 3. Клиент вызывал Edge Function с anon key как Bearer

**Риск:** Запросы к `/functions/v1/vault` шли с `Authorization: Bearer <anon>`, что не соответствует сессии пользователя и ломало корректную модель доступа после исправления функции.

**Меры:** `vaultHeaders()` подставляет `session.access_token` и заголовок `apikey` (anon key), как рекомендует Supabase.

### 4. Подмена `user_id` в теле запроса

**Риск:** Тело `POST` передавало `user_id` с клиента; при слабой проверке возможна подмена субъекта в аудите и бэкапах.

**Меры:** Идентификатор пользователя берётся только из проверенного JWT; поля `user_id` в теле для бэкапов / целостности удалены.

### 5. Секреты в `connection_config` для всех залогиненных

**Риск:** Политика `SELECT ... USING (true)` позволяла любому `authenticated` читать URL и ключи из `connection_config`.

**Меры:** Миграция `016_security_rls_tightening.sql` — чтение только для ролей owner/auditor (`is_owner_or_auditor()`).

### 6. Резервные копии и снимки видны всем пользователям

**Риск:** Любой пользователь видел список всех бэкапов и мог читать `backup_snapshots`.

**Меры:** Миграция 016 — видимость `created_by = auth.uid()` или staff; снимки привязаны к видимости родительского бэкапа.

## Средние (частично сняты или остаются как долг)

### 7. XSS через `innerHTML`

**Риск:** Имена файлов, продуктов, пути поиска, подписи в превью попадали в шаблоны без экранирования.

**Меры:** Массовое использование `esc()` для пользовательских и части серверных строк в `main.ts`. Рекомендуется периодически просматривать новые вставки в DOM.

### 8. Широкие политики RLS `USING (true)` на других таблицах

**Примеры:** `version_comments` SELECT для всех authenticated; `collab_sessions` read all; часть PLM-таблиц — чтение без привязки к владельцу продукта.

**Статус:** Не менялось в этом раунде; следующий шаг — сузить по `product_id`/участникам или по ACL узлов документов.

### 9. CORS `Access-Control-Allow-Origin: *` на Edge Function

**Риск:** Любой сайт может вызывать API с браузера пользователя, если тот залогинен (cookie-less JWT всё равно нужно украсть).

**Рекомендация:** В продакшене ограничить origin через переменную окружения.

### 10. Операции backup/restore в Edge Function с service role

**Статус:** Создание/верификация/восстановление по-прежнему используют service role после проверки роли **owner** в коде функции. Это осознанный компромисс для полного снимка БД; восстановление остаётся высокорисковой операцией — доступ только owner.

## Рекомендации на будущее

- Включить **JWT verification** на уровне шлюза Supabase (если доступно) в дополнение к коду функции.
- Разделить **секреты** (`connection_config`) и несекретные настройки по разным таблицам/столбцам.
- Пройтись по **остальным `innerHTML`** и вынести рендер в DOM API или шаблонизатор с автоэкранированием.
- **Зависимости:** периодически `npm audit`; для Deno-функций — проверка lockfile/импортов.

## Применение миграции 016

После `supabase db push` или применения SQL вручную убедитесь, что у хотя бы одного пользователя в `user_profiles.role = 'owner'`, иначе никто не прочитает `connection_config` из UI.
